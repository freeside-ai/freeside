# Saddle app bootstrap (#70)

## Decisions

- **Use Apple Swift OpenAPI Generator at build time, pinned to 1.13.0.** It consumes the repository's OpenAPI 3.0 contract natively, generates both public types and a transport-independent client, and keeps generated source out of review while the schema remains provisional. The runtime, URLSession transport, and HTTP types are also exact-version package dependencies, with `Package.resolved` committed. Rejected: OpenAPI Generator's Swift client, because it brings a separate template/runtime model and duplicates networking policy; hand-written Codable request types, because they would immediately create a second contract; committed generated Swift, because this build plugin makes it unnecessary churn.
- **Keep a mechanically refreshed schema mirror inside the Swift target.** SwiftPM build-tool plugins discover `openapi.yaml` in their target directory. The refresh command copies `api/openapi.yaml`, builds the generated client, and fails if the copy changed, proving a clean regeneration from the merged contract. Rejected: a symlink out of the target, because package/plugin sandbox behavior differs between `swift build` and Xcode; a second editable schema, because drift would be silent.
- **Expose one local Swift package to two thin native application targets.** The package owns generated API and shared SwiftUI code; the Xcode project owns only macOS/iOS app entry points and bundles. This keeps client behavior testable with `swift test` while producing real platform targets. Rejected: two projects with duplicated source membership; a SwiftPM executable called an iOS app, because it does not establish an application bundle target.

Revisit when the API ceases to be provisional or Apple Swift OpenAPI Generator's package-plugin contract changes; either may justify committing generated sources or replacing the schema-mirror step.
