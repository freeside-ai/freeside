# App CI runtime and cost

## Decisions

**Package verification runs on Linux in parallel with Apple-platform builds.**
Chose a Linux job for schema generation and Swift package tests over running the
entire gate serially on macOS: the API and its tests have no Apple-only imports,
while the macOS and iOS Xcode builds continue to compile the full package under
the supported Apple toolchain. The first hosted run disproved the broader
assumption that a test target's dependency closure limits `swift test`'s build
plan: SwiftPM also built the unrelated `FreesideCore` library and failed on its
SwiftUI import. Chose a manifest-level Linux graph containing only
`FreesideAPI` and `FreesideAPITests` over source-level conditional compilation;
the latter would pretend the SwiftUI product has a meaningful Linux form. This
preserves all four checks, shortens the critical path, and moves package-test
minutes to the lower-cost runner class. A failure on either job fails the gate.

**The iOS CI build targets the runner's active architecture.** Chose
`ONLY_ACTIVE_ARCH=YES` for the simulator build over the generic destination's
universal arm64 and x86_64 output: CI verifies source compatibility and linkage,
not distribution of a simulator binary, and the arm64 macOS 26 runner is the
supported execution environment for this gate. The macOS and iOS builds also
disable editor-only index generation and require the committed package
resolution.

**Compiled build outputs remain uncached.** Rejected caching SwiftPM `.build` or
Xcode DerivedData in this unit: the measured local DerivedData alone is about
540 MB, and restoring compiled state would weaken the clean-build signal while
adding cache-storage and toolchain-invalidation policy. Cancellation of
superseded runs captures wasted work without changing build provenance.

Revisit when the Swift package gains an Apple-only dependency, simulator
x86_64 becomes a supported CI requirement, or clean-build timing exceeds the
project's acceptable feedback budget after this split.
