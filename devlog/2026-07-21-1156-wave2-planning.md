## Decomposition

One strictly sequential, spine-owned chain, milestone 1A, each unit
blocking the next, targeting the internal exits in order:

#235 (1A.0 integration: engine + daemon binary over signet and the
fake StageDriver) → #41 (real-process kill matrix) → #182 (live
automation trust audit, `kind:contract`) → #236 (1A.1 integration:
fake candidate to published PR under the reviewed trust profile) →
#237 (1A.2: Claude driver, ward gate, `freesided submit`, real items)
→ #238 (operational packaging to the §10 targets).

Maintainer prerequisites filed as `needs-human`, no lane, unmilestoned
per the Pickup rules: #232 (first-repository selection, §11 boring
criteria), #233 (GitHub App via the manifest flow), #234
(trust-profile generation and one-time review). They enter units'
Dependencies fields; fiat assigns follow-up after the maintainer acts.

Consequential choices, with the alternatives rejected:

- **A single sequential chain, no parallel lanes.** Wave 2 is
  convergent integration (plan §11): every unit drives the same engine
  and harness paths, so parallelism would buy conflicts, not
  throughput. Rejected: running #182 concurrently with #235/#41 —
  contract units are exclusive anyway, and the chain keeps exactly one
  claimable unit at a time.
- **#41 as its own unit, second in the chain.** It carries a distinct
  escalated acceptance (the real-process kill matrix #34 could not
  meet) and its own declared paths. Rejected: folding it into #235,
  which would dilute the 1A.0 exit-verbatim acceptance and make the
  1A.0 gate depend on harness work the exit list doesn't name.
- **#182 scheduled mid-chain rather than left dormant.** #169's drift
  gate fails closed when no current `WorkflowAudit` exists, so
  publication to the first managed repository (#236) is inert without
  the live producer and the profile allow-axes. Placed after #41 and
  before #236; its GitHub API scopes are coordinated on the App issue
  (#233). Rejected: deferring it past 1A.1 with a fixture audit —
  that would demonstrate the 1A.1 exit against a hand-written audit,
  exactly the assumed-rather-than-proven pattern the Wave 1 audit
  flagged (#165, #170).
- **Packaging split out of the 1A.2 unit (#238 after #237).** §10 is
  explicit: build the installer only after the underlying interfaces
  survive real use. #237's acceptance is the §11 1A.2 exit minus its
  packaging bullet; #238 carries that bullet with the §10 clean-VM
  targets as acceptance. Rejected: one combined unit — its acceptance
  would mix "several real items completed" with VM-provisioning
  verification, and a packaging failure would reopen a proven exit.
- **Integration-bug rule (Template F) recorded on #231.** An
  integration bug traceable to one lane's package becomes a
  lane-labeled `kind:fix` issue, never an inline fix, unless trivial;
  domain/migrations/api changes stay `kind:contract`. This keeps the
  Wave 1 lane ownership boundaries intact through convergence.
