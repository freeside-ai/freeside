`/sync/bootstrap`, the attention/runs/conversations read surfaces,

## Decisions

- **OpenAPI 3.0.3, not 3.1.** The likely Go generator when the daemon
  grows a server (oapi-codegen) supports 3.0 only; Apple's
  swift-openapi-generator handles both. `nullable: true` covers the
  domain's pointer fields. Rejected: 3.1 (nicer null typing, but it
  would force either a different Go generator or a spec migration at
  exactly the moment the spec is first exercised). Provisional status
  keeps this a two-way door.
- **Validator: vacuum, pinned as `go run
  github.com/daveshanley/vacuum@v0.29.9`.** The repo's only toolchain
  is Go and CI already sets it up; a Go-module-pinned tool resolves
  deterministically, matching the pinned-golangci-lint convention.
  Rejected: `npx @redocly/cli` (needs no setup on GitHub runners but
  floats transitive dependencies without a lockfile, and adds a second
  language runtime to the verification story).
- **Per-type snapshot envelopes.** §5.14 requires every
  ResourceSnapshot to carry `as_of_revision` and `entity_version`;
  OpenAPI 3.0 has no generics, so the spec defines explicit
  `AttentionItemSnapshot`, `AttentionDeliverySnapshot`, `RunSnapshot`,
  `ConversationSnapshot` (boring, greppable). The wrapped entity
  property is named for the entity (`item`, `run`, ...), not a uniform
  `resource`.
- **Every mirrored field is required; pointers are nullable.** The
  daemon marshals domain types without omission (no `omitempty`), so
  presence-of-all-fields with `nullable: true` on Go pointer fields is
  the exact wire contract, not a style choice. Durations are int64
  nanoseconds (`submit_to_first_open`), timestamps RFC3339
  `date-time`, IDs and digests opaque non-empty strings.
- **ClientCommand carries exactly its five §5.14 fields**, enforced
  with `additionalProperties: false` (the acceptance says "exactly").
  `payload` stays a free-form object and `expected_bindings` a
  name→digest string map, both marked provisional: typing them now
  would invent command vocabulary ahead of the signet/saddle pair
  exercising the surface (no premature abstraction; widening is a
  later `kind:contract` change). Optimistic-concurrency rejection is a
  409 carrying the replacement item snapshot (sync test 2); idempotent
  retry semantics are stated on the operation (test 4); attachment
  upload convergence-by-digest on its operation (test 10).
- **Read surface scoped to the contract's checklist.** Findings,
  classifications, and resolved policies have domain types but no
  endpoints; devices/pairing likewise. They join the spec when a
  consumer needs them, as contract changes.
- **Entity examples are lifted verbatim from the domain golden files**
  so vacuum's `oas3-valid-schema-example` rule mechanically
  corroborates the field-for-field cross-check (mutation-tested: an
  invalid enum in an example fails the lint non-zero). This is a
  point-in-time copy, not a live link: if a domain golden changes, the
  spec example must move with it, which is exactly the coupled
  contract change the one-work-unit rule wants visible.
- **Every endpoint requires a provisional paired-device credential**
  (from Codex review, accepted): §5.14 makes device credentials and
  revocation part of the boundary (a revoked device cannot submit,
  sync tests 15-16), so a spec with no `security` models the surface
  as unauthenticated and generated handlers would inherit that. A
  bearer `deviceCredential` scheme, marked provisional, is required
  globally; the concrete token format and the pairing endpoints land
  together via a later contract change. Same review round: the
  non-empty-when-present class swept mechanically against the domain
  validators (`expected_bindings` values typed as `Digest`,
  `minLength: 1` on `run_id`, `conversation_id`, `source_head_sha`,
  `verification_recipe_digest`); `pr_head_sha` deliberately stays
  unconstrained because the domain does not reject an empty one.
- **Enum-branch admissibility invariants are encoded as oneOf branches**
  (from Codex round 2, accepted): the generic shapes admitted values the
  domain validators reject, and generated validators would inherit the
  laxness. Three restructurings: `evidence_snapshot` now references
  `EvidenceArtifact`/`EvidenceProvenance` (producer_class restricted to
  verifier/daemon, recipe digest required, mirroring
  `EligibleForEvidenceSnapshot`; an agent artifact is unrepresentable as
  evidence, and the `agent` producer class now appears nowhere on the
  wire since agent output reaches clients only as labeled agent_claims);
  `AttentionDelivery` is a status-discriminated oneOf whose three
  branches encode the receipt-correspondence rule (a null-only receipt
  is `nullable: true, enum: [null]`); and, sweeping the class beyond
  the cited findings, `Subject` splits into run-scoped and unscoped
  branches so a project/system subject with a run_id is
  unrepresentable. The standalone `SubjectType`/`DeliveryStatus`/
  `ProducerClass` enum schemas dissolved into the branches. **Class
  boundary drawn deliberately**: enum-discriminated value admissibility
  is encoded; cross-field temporal/derived relations (receipt ordering,
  the whole of TimingSummary.Validate) are not schema-expressible in
  3.0 and stay prose notes on the schemas, enforced by the domain
  validators. Branch encoding mutation-tested (an opened example
  relabeled `submitted` fails the lint).
- **Remaining domain Validate constraints swept into the schema**
  (from Codex round 3, two P2s, accepted): `requested_decision` gains
  `minItems: 1` (domain `ErrNoActions`) and `AgentClaim.label` gains
  `minLength: 1`. The cited class re-enumerated one level wider, as
  "every schema-expressible domain.Validate constraint on a wire
  type", by walking every Validate function: that added
  `EvidenceArtifact.type` `minLength: 1` (artifact.go rejects an empty
  type) and `minimum: 0` on `submit_to_first_open` (a derived but
  always-true bound: opened is on or after submitted and the duration
  must equal their difference). Checked not to over-constrain:
  `reason`, `pr_head_sha`, and `Message.body` stay unconstrained
  (domain accepts empty), `Conversation.messages` and `Run.stages`/
  `Stage.attempts` allow empty lists (domain does). Both cited
  constraints mutation-tested (empty `requested_decision` and empty
  `label` in the example fail the lint).
- **Round 4 (two P2s, accepted): evidence publish_eligible pinned to
  true, and an attachment read endpoint.**
  `EvidenceArtifact.publish_eligible` becomes `enum: [true]`: the
  evidence gate (`domain.EligibleForEvidenceSnapshot`) admits an
  artifact only when the approved-recipe computation succeeds, and that
  same computation drives publish_eligible, so an admitted evidence
  artifact always carries true; recipe approval itself stays runtime
  policy, but its consequence is schema-expressible (mutation-tested:
  `publish_eligible: false` in the example fails the lint).
  `GET /attachments/{digest}` added: the spec's own conversation prose
  said attachments "are fetched by digest" while defining no fetch
  operation, and plan §4 has cards rendering image attachments from the
  artifact store by digest, so the read path was a genuine internal
  inconsistency, not deferred scope; the digest path parameter hoisted
  to the path item, shared by GET and PUT. This is one endpoint beyond
  the contract's Acceptance-2 checklist, added by review the same way
  round 1 added the security scheme.
- **Two recommended vacuum rules disabled, documented in
  `api/vacuum.ruleset.yaml`**: `camel-case-properties` (the wire
  format is snake_case by design, mirroring domain JSON tags) and
  `oas3-missing-example` (it demands examples on every media type and
  property; the load-bearing rule is example validity, which stays
  on). With those two off, CI runs `--fail-severity warn`, so the spec
  is held at zero findings rather than zero errors.
